2-6. Objects

bags of data

2-6-1

This section is an overview of objects, in case you need a refresher!

2-6-2

JavaScript has two composite data types, namely objects and arrays!

Althought technically arrays are a special kind of objects, but, never mind.

2-6-3

An object in JavaScript is a collection of key-value pairs, often called a dictionary in other languages.

The property names (keys) are always strings, and the property values can be anything.

2-6-4

We can create an object literal like this:

var myObject = {
  someProp: 42,
  anotherProp: "hello!"
};
2-6-5

There are two different ways to access a value.

We can use dot notation:

var x = myObject.someProp; // 42
2-6-6

Or we can use bracket notation:

var y = myObject["someProp"]; // 42
2-6-7

Question: The dot notation seems much smoother, so why would we ever want bracket notation?

2-6-8

Answer: That allows us to do dynamic lookups:

var propName = "someProp";
var z = myObject[propName]; // 42
2-6-9

We add a property simply by assigning to it:

var myObject.newProp = 777;
2-6-10

And we delete a property using the delete keyword:

delete myObject.newProp;
2-6-11

If we access a non-existing property we always get undefined.

myObject.bogusProp; // undefined
2-6-12

Contrary to primitives, objects are references, which means that this is false since it is two different objects:

var test = ( {a:1} === {a:1} ); // false
2-6-13

While this is true, since they point to the same object:

var x = {a:1};
var y = x;
var test = x === y; // true
2-6-14

Since objects are references they are always truthy:

if ({}){
  console.log("An empty object is truthy!"); // will be shown!
}
2-6-15

We can iterate over objects using a for-in loop:

for (var key in myObject){
  console.log(key+" has value "+myObject[key]);
}
2-6-16

Finally: object can contain other objects:

var me = {
  name: "David",
  address: {
    street: "Åsgatan 9",
    zip: 27131,
    city: "Ystad"
  }
}
2-6-17

We can access properties to any depth: 

var myZip = me.address.zip;

...although we might have to guard for undefineds:

var myZip = me.address && me.address.zip;

2-8. Functions

code in a can

2-8-1

This section is an overview of functions, in case you need a refresher!

2-8-2

There are two different ways to define a function in JavaScript. First as a named function:

function echoer(word,times){
  var echo = "";
  for(var i = 0; i < times; i = i + 1){
    echo = echo + word;
  }
  return echo;
}
2-8-3

Or as an anonymous function which we can assign to a variable:

var echoer = function(word,times){
  var echo = "";
  for(var i = 0; i < times; i = i + 1){
    echo = echo + word + " ";
  }
  return echo;
}

Mostly it doesn't really matter which way you choose.

2-8-4

As the example showed, we use return something to return a value. If you just do return;, the returned value is undefined, which it also is if you don't use return at all.

2-8-5

We execute a function by using parens, and listing eventual parameters within those:

var withEcho = echoer("hello",3); // "hello hello hello "
2-8-6

Question: What is the difference between this:

var x = func;

and this:

var x = func();
2-8-7

Answer: Well:

  • In the first example, x has a reference to func. Which means that after that, doing x() and func() would yield the same thing.

  • In the second example, x is whatever func returned.

2-8-8

A final observation - There is nothing stopping us from having functions on objects:

myObject.beingAnnoying = function(){
  console.log("SPAM!");
};
myObject.beingAnnoying(); // "SPAM!"

When a function is a property on an object it is called a method of that object.

2-8-9

And to end on a meta note; there is also nothing stopping us from having objects (or anything else) on functions, since functions are objects.

var func = function(){};
func.prop = {hello:"world!"};
console.log(func.prop.hello); // "world!"

2-10. Functional programming

Jumping down the rabbit hole

2-10-1

JavaScript is a functional language, which modern JS makes frequent use of.

Let's just do a quick sanity check to make sure we're all on the same page!

2-10-2

So - since functions are first class citizens, we can send them around just like any value.

Which also means that a function can take, and/or return, other functions! Such a function is called a higher order function.

2-10-3

As a contrived example, say we have this function:

var spam = function(){
  console.log("SPAM!");
}
2-10-4

And then we have this higher order function:

function repeater(func,times){
  for(var i = 0; i < times; i = i + 1){
    func();
  }
}
2-10-5

If we invoke repeater like this:

repeater(spam,3);

We would see this in the console:

"SPAM!"
"SPAM!"
"SPAM!"
2-10-6

To show we can also return new functions, take a look at this beauty:

function multiplier(func,times){
  return function(){
    for(var i = 0; i < times; i = i + 1){
      func();
    }
  };
}
2-10-7

Did you see the difference? multiplier doesn't execute the parameter function, but returns a new function!

var tripleSpam = multiplier(spam,3);

If we execute the returned function we get the triple spam:

tripleSpam(); // SPAM! SPAM! SPAM!
2-10-8

Functional programming is a really powerful tool, and something that is likely to get you hooked once you have learned it. We warmly encourage you to explore the subject!

2-10-9

Note: the functional paradigm is about much more than just passing functions around, we've really just scratched the surface here.

2-11. map, filter and reduce

the cornerstones

2-11-1

For applied functional programming, there are three list-operating tools that have a very central position:

  • aThe map method
  • bThe filter method
  • cThe reduce method

Let's take a look at these!

2-11-2

First, map. It takes a list and an iterator function as arguments, and returns a new list.

2-11-3

The iterator function that we pass in is called with each element and the index of that element, and it should return a new element to be used instead:

2-11-4

Here's an example. What will this call return?

var list = ["David", "Carl", "Eric"];
var newList = list.map(function(elem){
  return elem+"y Mc"+elem+"face";
});
2-11-5

Yup, newList will now equal this:

["Davidy McDavidface","Carly McCarlFace","Ericy McEricface"]
2-11-6

In essence, .map will create a new list of the same length, where each element is individually transformed by the iterator.

2-11-7

Next up is .filter, which is probably the easiest to understand. It has the exact same signature as .map:

2-11-8

The iterator is similar too, except it returns a boolean saying whether or not the element should go into the new array.

2-11-9

Now for .reduce! It is more powerful, but also harder to grasp.

2-11-10

Similar to .map it takes a list and an iterator, but also a seed value. Unlike .map it doesn't necessarily return a list, instead it can return any value.

2-11-11

The iterator function is called with each element and the current value (sometimes called memory or accumulator), and returns a new value.

2-11-12

The seed that we passed in to .reduce is used as the current value for the first iterator call.

The result of the .reduce call is whatever is returned from the last iterator call.

2-11-13

In other words, it looks like this (for a 3-item list):

2-11-14

Here's a code example. What does it return?

var list = ["David", "Carl", "Eric"];
var something = list.reduce(function(mem, elem){
  mem[elem.length] = mem[elem.length] || 0;
  mem[elem.length]++;
  return mem;
}, {})
2-11-15

it returns an object which counts how many words of each length the list contains:

{
  4: 2,
  5: 1 
}

4-2. DSL

4-2-1

In the prior example, you saw how templates have data placeholders where it puts the given variables.

The fancy term for this is interpolation, btw.

4-2-2

But, this is not enough! Consider for example a view that is supposed to render a bunch of links to a list of posts, which might be empty. We need to...

  • repeat some HTML for every link
  • or conditionally display an apologetic message if there are no posts yet.
4-2-3

In order to do that we must have some kind of looping, as well as a conditional.

But since HTML isn't a programming language, it doesn't have any of that!

This means that the templating solution must provide this too.

4-2-4

Here's a Handlebars solution to the aforementioned problem:

<div class="posts">
  <h2>Posts</h2>
  {{#if posts.length}}
  {{#each posts}}
  <a class="post" href="{{this.url}}">{{this.title}}</a>
  {{/each}}
  {{else}}
  <p>No posts :(</p>
  {{/if}}
</div>
4-2-5

Picking this apart we see that we do loops like this:

{{#each list}}
  <p>Everything in here is repeated for every item in list</p>
  <p>And {{this}} points to the current item</p>
  <p>Also, {{@index}} is the current index (0,1,2...)!</p>
{{/each}}
4-2-6

And here's the syntax for conditionals:

{{#if someVar}}
  <p>I will only be shown if someVar is truthy</p>
{{/if}}
4-2-7

In other words, Handlebars has invented it's own words for loops and conditionals (and other things too).

The fancy term for this is a DSL, short for a Domain Specific Language.

4-2-8

This means that whenever we learn a templating system, we will likely also have to learn a DSL.

AngularJS is no different. In fact, since so much is controlled from the template, it probably has a more complex DSL than most frameworks.

5-1-1

Like animals!

...why is it more complex than ng-repeat?

Because we need to say two things about each option:

  • what value should it have
  • what text should be displayed
<select ng-model="selectedFruit" ng-options="f.name for f in fruits"></select>

5-2. Filters

including a filter named filter

5-2-1

Remember the debug tip from before?

So, what is that pipe thing there?! That isn't regular JavaScript!

5-2-2

That's called a filter, and it is a special syntax AngularJS introduced. The filters are really just normal functions.

There are many more filters than json, and now we'll meet two:

  • aorderBy
  • bfilter
5-2-3

First, orderBy! This is a useful filter to control sorting of a list of objects.

Say we expose this list from the controller:

$scope.list = [
  {name: 'David', age: 37},
  {name: 'Jacob', age: 30}
]
5-2-4

If we render them in an ng-repeat...

<div ng-repeat="person in list">
  {{person.name}}
</div>

...we'll get David first, then Jacob, since that's their order in the list.

5-2-5

But if we use orderBy and pass in 'age' as a parameter:

<div ng-repeat="person in list | orderBy : 'age' ">
  {{person.name}}
</div>

...then we get Jacob first, and then David!

(note the weird colon syntax for parameters)

5-2-6

The orderBy filter can also take a second parameter - if that is truthy, then the order will be descending instead. So this...

<div ng-repeat="person in list | orderBy : 'age' : true">
  {{person.name}}
</div>

...would mean we get David first again!

...and rendered them like this:

As before, the true power is of course if we made the filter string dynamic!

6-1. Versatile object definitions

defining objects like a boss

6-1-1

In ES2015 we got five small but nice features for defining objects in a smoother way:

  • adynamic keys
  • bautomatic same-key-value
  • cmethod shorthand
  • dgetters
  • esetters
6-1-2

If we wanted to create an object with a dynamic key we had to go about it in a roundabout way before:

var obj = {};
obj[dynamicKey] = someValue;
6-1-3

Now, instead, we can use the dynamic key syntax by wrapping it in brackets:

var obj = {[dynamicKey]: someValue};
6-1-4

Also, if our value is in a variable with the same name as the intended key, like here:

var person = {
  name: name,
  age: age
};
6-1-5

...ES2015 introduces a shorthand syntax:

var person = {name, age};
6-1-6

And if we define an object with a method:

var obj = {
  method: function(arg1,arg2){
    // do advanced stuff
  }
};
6-1-7

...ES2015 lets us be less verbose by using the method shorthand syntax:

var obj = {
  method(arg1,arg2){
    // do advanced stuff
  }
};
6-1-8

This can also be combined with the dynamic key syntax:

var obj = {
  [methodName](arg1,arg2){
    // do advanced stuff
  }
};
6-1-9

Finally, ES2015 also introduced getters and setters.

Let's look at getters first. They are very useful for dealing with computed properties.

6-1-10

Say we're working with user objects like this:

var user = {
  firstName: "John",
  lastName: "Doe"
};

Now we want to implement a computed property fullName.

6-1-11

Here's an ES3 solution doing it as a method:

var user = {
  firstName: "John",
  lastName: "Doe",
  fullName: function(){
    return this.firstName + ' ' + this.lastName;
  }
}

user.fullName(); // John Doe
6-1-12

By using an ES2015 getter we can access the computed property normally instead:

var user = {
  firstName: "John",
  lastName: "Doe",
  get fullName() {
    return this.firstName + ' ' + this.lastName;
  }
}

user.fullName; // John Doe, without invocation!

Cf. uniform access principle.

6-1-13

A setter let's you act upon prop mutation, for example logging...

var user = {
  set userName(str) {
    log(this._userName + " changed name to " + str);
    this._userName = str;
  }
}

user.userName = "Steve"; // "Bob changed name to Steve"
6-1-14

...or validation:

var user = {
  set userName(str) {
    if (str.match(/[^a-z]/)){
      throw "Name can only contain lowercase letters!";
    }
    this._userName = str;
  }
}

user.userName = "Bob the 1 and only"; // Error: Name can only contain..
6-1-15

Did you note that we used a different property name inside the setter? The setter was for userName, but inside it we instead set _userName.

Why do you think that is?

6-1-16

If we mutated the same property inside the setter then that would trigger the setter to be called, which would mutate the property, which would trigger the setter, etc. We would end up in an infinite loop.

6-1-17

Sometimes it makes sense to have setters for computed properties*:

var user = {
  get fullName() {
    return this.firstName + ' ' + this.lastName;
  },
  set fullName(str) {
    this.firstName = str.split(' ')[0];
    this.lastName = str.split(' ')[1];
  }
}

6-2. Destructuring and rest

cherry-picking the raisins from the cookie

6-2-1

Destructuring is a way to pick values from nested structures without having to do the manual digging.

6-2-2

Let's say we have an array of contenders, each represented by an object.

let contenders = [
  {name: "David", age: 37},
  {name: "Carl", age: 38}
  /* and a few others */
];

They are sorted by position so the first contender won, etc.

6-2-3

If we wanted the name of the winner we would do something like this in ES5:

let winnersName = contenders[0].name;
6-2-4

With destructuring, we can instead do this:

let [{name: winnersName}] = contenders;

Or, combined with the same-key-value shorthand:

let [{name}] = contenders;
6-2-5

Destructuring also allows us to use the powerful rest element which can lump up many array elements into one, making for some very succinct code:

let [winner, ...losers] = contenders;
6-2-6

Note that the rest element has to be the last one in the array, so this wouldn't work:

let [...others, superloser] = contenders; // syntax error
6-2-7

Wait.. Theoretically, the rest could be placed anywhere, as long as there's just one. The parser should still be able to figure out what's what!

Right?

6-2-8

True. But that would require lookahead, which is complex and more taxing. And so the choice was made to only allow the rest element in the last position.

6-3. Versatile function definitions

defining function like a boss

6-3-1

ES2015 provides several neat features for defining functions:

  • adefault parameter values
  • brest parameters
  • cdestructuring parameters
  • darrow functions
6-3-2

Default parameter values exist in many languages, and was popularised in JS through CoffeeScript.

The idea is to handle optional parameters in a smoother way.

6-3-3

Creating a function with an optional parameter in ES3 meant we had to do a sometimes tedious dance of initialization:

function makePerson(name, age) {
  var age = age || 'unknown';
  // do complex stuff
}
6-3-4

With default parameter values we can instead do this: 

function makePerson(name, age = 'unknown') {
  // do complex stuff
}
6-3-5

The second new feature, rest parameters, is a way of capturing multiple arguments into a single variable like a rest element in a destructuring.

This can often save us from having to do awkward stuff with the not-quite-an-array arguments object.

6-3-6

Imagine a competition function that is called with all contenders one by one:

function competition() {
  var contenders = Array.prototype.slice.call(arguments);
  var winner = contenders[0];
  var losers = contenders.slice(1);
  // do something with winner and losers
}
6-3-7

Using rest parameters, this function simply becomes:

function competition(winner, ...losers) {
  // do something with winner and losers
}
6-3-8

Note that the rest parameter has to be the last parameter, just like the rest element, and for the same reason.

6-3-9

Remember destructuring? We can use that in signatures:

function introduce({name, age}) {
  console.log(name,"is",age,"years old");
}
var me = {name: "David", age: 37};
introduce(me); // David is 37 years old
6-3-10

Finally - know how defining anonymous functions in JS is rather verbose?

var mcboatify = function(arg) {
  return "Boaty Mc"+arg+"Face";
};
6-3-11

With arrow functions things feel less heavy:

var mcboatify = (arg) => {
  return "Boaty Mc" + arg + "Face";
};
6-3-12

They can become smaller still - if we have exactly one parameter, we can omit the parenthesis in the signature:

var mcboatify = arg => {
  return "Boaty Mc" + arg + "Face";
};
6-3-13

Finally, if you just want to return an expression, we can skip brackets and the return keyword:

var mcboatify = arg => "Boaty Mc" + arg + "Face";

Now the function body consists of a single expression, which will be implicitly returned.

6-3-14

Note however that if you want to use the single expression form with an object literal, we have to wrap it in parenthesis to distinguish it from a regular function block:

var createUser = (name,age)=> ({name,age})
6-3-15

Arrow functions are not only less heavy to write, they are also lighter for the interpreter since they don't get an implicit context parameter.

6-3-16

Which means that if you refer to this inside an arrow function, it is the same this as on the outside.

var me = this;
setTimeout(() => {
  console.log(this === me); // true
}, 10);
setTimeout(function(){
  console.log(this === me); // false
}, 10);
6-3-17

As a final note; arrow functions can beautifully describe the flow for nested higher order callbacks. Take this convoluted old code:

function multiplier(func,times){
  return function(){
    for(var i = 0; i < times; i = i + 1){
      func();
    }
  };
}
6-3-18

With arrow functions, that becomes:

var multiplier = (func,times)=> ()=> {
  for(var i = 0; i < times; i = i + 1){
    func();
  }
}

6-4. Spreads

the dark side of rests

6-4-1

You have already seen how we use rest element/parameter to capture several array elements into a single variable:

var [winner, ...losers] = competitors;
6-4-2

Now imagine the opposite scenario - we have the winner and losers variables, and want to define competitors. In ES3 this is done like this:

var competitors = [winner].concat(losers);
6-4-3

ES2015 gives us a new options - spreads! It looks exactly like rest, but we use it on the right side instead (or when we call a function as opposed to when we define it):

var competitors = [winner, ...losers];

We say that we spread the contents of the expression into the outer array.

6-4-4

Spreads gives us a less verbose way to copy an object and add properties to it, which is otherwise done like this:

var augmentedObj = Object.assign({}, oldObj, newProps);
6-4-5

With spreads we can instead do this:

var augmentedObj = {...oldObj, ...newProps};
6-4-6

Note that while spreads and rests with arrays are in the spec for ES2015, object spread is still a Stage 3 proposal (April 25th, 2017).

It is expected to be accepted into an upcoming release of the language, and is already supported by Babel and the like.

6-5. Modules

getting into the import/export business

6-5-1

Node gave us modules through the require and module.exports globals it provides.

But with ES2015, we got native modules for the very first time!

6-5-2

While Node modules followed the CommonJS module standard, what was implemented in the language follows another syntax, named ES modules.

But the concepts are the same. While you would do this in CommonJS...

// file1.js
module.exports = {..};

//file2.js
var lib = require("./file1.js");
6-5-3

...you would do this with ES modules:

// file1.js
export const lib = {..};

//file2.js
import {lib} from './file1.js'

We have to name our exports here, otherwise things are pretty similar.

6-5-4

ES modules also allow for a default export:

// file1.js
export default const lib = {..};

//file2.js
import lib from './file1.js'

Note how we're not using curly braces now.

6-5-5

ES modules have an important limitation - they must be used in the top-level scope.

So dynamic stuff like this won't work:

if (someCondition){
  import lib from './file1.js';
}
6-5-6

Isn't that really limiting, and a big step back from .require?

6-5-7

Yes it is limiting, and no it is a gigantic step forwards. This allows us to do static analysis of module dependencies, and for example build tools can do clever things with that information.

6-5-8

There are other differences too, so for the full scope you should check the MDN docs for import and export

6-6. Classes

Waiter, there are classes in my JS!

6-6-1

Before ES2015, JavaScript used to famously lack classes.

This was not an oversight. Consider what classes are normally used for:

  • resusing functionality and
  • setting up hierarchies
6-6-2

In JavaScript this is addressed by

  • simply grabbing methods and/or mixing objects
  • prototypal "inheritance", which should really be called delegation
6-6-3

This means that classes didn't really serve a purpose. Yet they were still frequently used, through the weird, bolted-on new syntax which makes functions behave like constructors:

var user = new User("David", 1979);
6-6-4

But to really make this behave like normal classes...

var lucas = new Dog("Lucas");
lucas instanceOf Dog; // true
lucas instanceOf Animal; // true
lucas.bark(); // Lucas goes woof!
6-6-5

...then lots of jumping through hoops had to be done:

Dog.prototype = new Animal();
Dog.prototype.constructor = Animal;
Dog.prototype.bark = function() {
  console.log(this.name, "goes woof!");
}
6-6-6

To facilitate "class" use in JavaScript, ES2015 introduced the class syntax:

class Dog extends Animal {
  bark() {
    console.log(this.name, "goes woof!");
  }
}

Note how method shorthands are available in class declarations too!

6-6-7

But it is important to note that this does not mean that JavaScript has "normal" classes.

Under the hood the same weird prototype and constructor dance happens.

6-6-8

Still, since the syntax hides the mismatch, it can be a convenient way to package functionality. Specifically:

  • aconstructor
  • bmethods
  • cproperties
6-6-9

First off, what used to go in the fake constructor...

function Animal(name) {
  this.name = name;
}
6-6-10

...is now placed in a literal constructor method in the class declaration:

class Animal {
  constructor(name) {
    this.name = name;
  }
}
6-6-11

If you want the inherited constructor to be invoked too, you must do so yourself with the new super keyword:

class Dog extends Animal {
  constructor(name) {
    super(name);
    this.nickname = name + 'y boy';
  }
}
6-6-12

And you've already seen methods:

class Dog extends Animal {
  constructor() { ... }
  bark() {
    console.log(this.name,"goes woof!");
  }
}
6-6-13

Finally, as you saw, properties are normally initialised in the constructor:

class Animal {
  constructor(name) {
    this.name = name;
  }
}
6-6-14

...although with TypeScript we can also initialise properties directly on the class declaration:

class Dog {
  numberOfLegs = 4;
}
6-6-15

Of course there is a proposal for this to come to JS too.

So far, the focus of the class spec has been to lay a good foundation. There are many ways in which classes will gain functionality in the future.

6-6-16

So, to recap:

  • classes are just a light syntactic sugar introduced in ES2015
  • we normally don't need them in JavaScript
  • but they are a convenient way to bundle related functionality
6-6-17

Another note: In other languages we associate classes strongly with using inheritance.

But in JavaScript, classes are mainly a state machine vehicle. Functionality reuse can be achieved in other ways.

6-6-18

And, of course, they're used to represent components in React, so there's no escaping them!

6-7. Decorators

dewhatnow?

6-7-1

The situation around decorators is rather confusing;

  • There is a proposal to add it as a language feature
  • There is a slightly different implementation in TypeScript
  • There are connections/overlap with Reflect
  • There is disagreement on whether decorators are a good idea at all
  • There is (was?) something called annotations that is sort of the same... yet not
6-7-2

Focusing on the TypeScript implementation which is the most common, decorators are a way of decorating a class...

  • declaration
  • property
  • getter or setter
  • method
  • method parameter
6-7-3

As a simple example, imagine that we have a debounce function that throttles other functions:

function debounce(fn) {
  // ... create a throttled version of `fn`...
  return throttledFn;
}
6-7-4

And then we have a class with a method that is very expensive to call:

class myClass {
  myExpensiveMethod: function() {
    // lots of heavy lifting here
  }
}
6-7-5

Without decorators we would do this:

class myClass {
  myExpensiveMethod: debounce(function() {
    // lots of heavy lifting here
  })
}
6-7-6

With decorators, instead, we use the @ syntax:

class myClass {
  @debounce
  myExpensiveMethod: function() {
    // lots of heavy lifting here
  }
}

The end result is the same thing.

6-7-7

We can also have decorators that take additional arguments. For instance debounce could accept a minimum number of milliseconds:

class myClass {
  @debounce(300)
  myExpensiveMethod: function() {
    // lots of heavy lifting here
  }
}

In other words, decorators are just a light syntax sugar.

6-7-8

If you want to dig deeper into decorators, check out...

  • The decorator section of the TypeScript handbook
  • This concice and clear explanation with examples and interactive links
6-7-9

Note that in babel, because of some ongoing confusion, decorators are disabled. To play with them you currently need a specific plugin:

"plugins": ["transform-decorators-legacy"]

6-8. Miscellaneous

odds and ends

6-8-1

There's three more things worth mentioning:

  • adeclaring variables with let
  • bdeclaring variables with const
  • ctemplate strings
6-8-2

Variables in JavaScript have functional scope.

Even if you declare them inside an if-block in the middle of a function, the variable is still visible throughout the entire function.

6-8-3

So when you write this...

function myFunc(arg,lib){
  if (arg === 42){
    var ret = lib.method() + 7;
    return ret;
  }
  // do sth else
}
6-8-4

...this is what (conceptually) happens:

function myFunc(arg,lib){
  var ret;
  if (arg === 42){
    ret = lib.method() + 7;
    return ret;
  }
  // do sth else
}

In other words, the declaration is hoisted to the top.

6-8-5

This is generally considered a design mistake, and can give rise to weird bugs.

ES6 therefore introduces let as an alternative to var for declaring variables, and the only difference is that let has block scope.

6-8-6

In most languages there's a way to define constants, meaning a variable that cannot change.

This is missing from JavaScript.

6-8-7

A common "hack" is to name constants in all capitals:

var SOME_CONST = 42;

But this has no technical significance, it is just a hint.

6-8-8

ES6 therefore introduces const as another alternative to var, and the only difference is that you cannot reassign the value.

const answer = 42;
answer = 43; // throws an error
6-8-9

Finally, template strings!

let userTempl = `
  First name: ${user.fname}
  Last name: ${user.lname}
`;
6-8-10

As you saw, template strings...

  • are defined inside two backticks
  • can contain linebreaks
  • allow interpolation inside ${}
6-8-11

There's also a semi-secret way to invoke functions with templates. Here's an example from Choo:

html`
  <main class="app">
    Count: ${state.counter.count}
    <button onclick=${(e) => send('counter:increment')}>+</button>
  </main>`

The html function is invoked with the templates and interpolated values.
